we're going to be overhauling the @packages/evals/ cli. let's plan together through the implementation. 

We want our evals to have 3 levels of abstraction. First level is plain, deterministic, perfomance tests/evals for mainly the understudy functions bundled in @packages/cli/. Second level: interpretability (by an ai coding agent, for instance claude code or codex) of the exposed functions through the cli (commands). The third is the closest of what we do today, have an agent be given a task or a suite of tasks with different combinations of model/harness/tools and evaluate the performance and accuracy of such tasks/benchmarks. 

We're going to mainly focus on the first and third layers for know, but don't lose sight on the proposed architectural changes of the second. Here are the requirements:
- Braintrust for concurrency/traces (as we're using)
- for tier 1: deterministic, no evaluator.
- for tier 3: evaluator
- Browserbase built tin support (as we're using)
- Improve the DX to add more tasks on either layer
- lastly, a TUI for 'evals' cli

Use askusertool to clarify things, I know this is an extremely high level overview so we have to go back and forth a few times to ensure you have a clear vision on the overhaul.

Your filled packages/evals/interview.md:13 changes the center of gravity
  of the system.

  The current spec in packages/evals/spec.md:9 is still useful for

  - latency

  The system should become a 3-layer stack:

  1. Browser Target
     This is just how you get a Chromium endpoint.
     Examples:

  - local
  - browserbase

  Its job is to produce a session handle plus CDP connection info.

  2. Tool Adapter
     This is the deterministic tool layer under test.
     Examples:

  - understudy_v3
  - playwright
  - raw_cdp
  - browse_cli

  Its job is to implement the same small command surface against a target.

  3. Agent Harness
     This is bench-only.
     Examples:

  - Stagehand agent
  - stagent-style skill harness
  - MCP-driven agent
  - future CLI agent harnesses

  Its job is to solve tasks using one tool adapter.

  That gives you the separation you want:

  - core = compare tool adapters
  - bench = compare full harness + tool + model stacks

  So the matrices become:

  - Core: target x adapter x task x trial_profile
  - Bench: target x adapter x harness x model x task x trial_profile

  That is the clean seam.

  tasks.

  Core tasks should receive a minimal deterministic contract, something

  interface CoreToolSession {
    goto(url: string): Promise<void>;
    reload(): Promise<void>;
    back(): Promise<void>;
    forward(): Promise<void>;

    click(selector: string): Promise<void>;
    clickCoordinates(x: number, y: number): Promise<void>;

• Short version: there isn’t one honest “core” denominator here, there are
  two.

  The first is CDP-grade browser control. Raw CDP exposes page lifecycle, t
  ab lifecycle, JS evaluation, screenshots, keyboard/mouse events, and view
  port emulation through the Page, Target, Runtime, Input, and Emulation do
  mains. Playwright’s direct Page/BrowserContext API covers the same territ
  ory at a higher level, and the MCP/CLI tools all expose variants of that
  same control plane. (chromedevtools.github.io
  (https://chromedevtools.github.io/devtools-protocol/tot/Page/))

  The second is agent-facing representation. Agent-browser centers snapshot
  + refs, Playwright MCP centers structured accessibility snapshots and ref-
  based actions, and Playwright’s own MCP docs explicitly contrast MCP with
  CLI+skills on token efficiency. Chrome DevTools MCP and dev-browser also
  expose snapshot-oriented agent workflows. That means “page representation
  for AI” is important, but it is not the same layer as raw browser control,
  and the snapshot/ref formats are not standardized across tools. (github.c
  om (https://github.com/vercel-labs/agent-browser))

  Locally, Understudy already shows the split. packages/core/lib/v3/
  understudy/context.ts:157 is a CDP attachment layer; packages/core/lib/v3/
  understudy/page.ts:800 exposes the primitive browser operations; packages/
  core/lib/v3/v3.ts:933 can also create and attach a Browserbase session
  itself. browse similarly owns local strategy resolution and Browserbase-
  vs-local startup before calling Stagehand init in packages/cli/src/
  index.ts:685.

  So the true base contract for core should be the primitive layer:

  - Session lifecycle: prepare/attach, close, listPages, newPage, selectPag
    e, closePage. Raw CDP, Playwright, agent-browser tabs, and Chrome DevTo
    ols MCP all support this shape. (chromedevtools.github.io
    (https://chromedevtools.github.io/devtools-protocol/tot/Target/))
  - Navigation: goto, reload, back, forward, url, title. (chromedevtools.gi
    thub.io (https://chromedevtools.github.io/devtools-protocol/tot/Page/))
  - Inspection: evaluate, screenshot. (chromedevtools.github.io
    (https://chromedevtools.github.io/devtools-protocol/tot/Emulation/))
  - Input: pressKey, typeText, click, hover. (chromedevtools.github.io
    (https://chromedevtools.github.io/devtools-protocol/tot/Input/))
  - Viewport and generic waits: resizeViewport, waitForLoadState, waitForTi
    meout, waitForPredicate. (chromedevtools.github.io
    (https://chromedevtools.github.io/devtools-protocol/tot/Emulation/))

  But the target model is not common. Some tools are selector/locator-first,
  some are snapshot-ref-first, and some are really coordinate/CDP-first. Un
  derstudy core today is visibly coordinate-heavy in packages/core/lib/v3/u
  nderstudy/page.ts:1424, while browse documents both selector/ref and coor
  dinate commands in packages/cli/README.md:57. Agent-browser is heavily re
  f/selector oriented, and Playwright MCP requires either a ref or selector
  for many actions. So click(selector) is the wrong base abstraction; it sh
  ould be click(target) with capability-gated target kinds like coords | se
  lector | snapshot_ref. (github.com
  (https://github.com/vercel-labs/agent-browser))

  Likewise, snapshot should not be in the strict base interface. It is comm
  on enough to deserve a first-class optional capability family, but not co
  mmon enough to define core’s entire contract. Agent-browser, Playwright M
  CP, Chrome DevTools MCP, dev-browser, Understudy, and browse all have sna
  pshot-like concepts, but they differ in shape, ref semantics, and intended
  use. For AI ergonomics, I’d make this a separate representation suite ins
  ide core, and compare raw payload size, token estimate, ref stability, and
  actionability there. (github.com
  (https://github.com/vercel-labs/agent-browser))

  The other correction is your launch/attach point: browser target cannot be
  fully abstracted away. Agent-browser owns a daemon and can connect via CDP
  or auto-connect; Playwright MCP supports persistent profiles, isolated mo
  de, shared browser context, extension connection, and CDP endpoint config;
  Chrome DevTools MCP can either start Chrome itself or attach via --browse
  rUrl / --wsEndpoint; dev-browser’s public docs also describe launching a
  Chromium relay. That means environment should stay a reporting axis, but
  the runner also needs target_ownership and connection_mode as first-class
  metadata because startup behavior is part of what you are evaluating. (gi
  thub.com (https://github.com/vercel-labs/agent-browser))

  So I’d model core as:

  - Base required capability family: session + page primitives
  - Optional capability families: selector-targeting, snapshot-
    representation, network, console, performance, uploads/dialogs
  - Separate run metadata: environment, target_ownership, connection_mode,
    startup_ms, warm_ms
